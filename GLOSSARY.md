# 用語集 (MasaCAD)

プロジェクト内で使う主要用語をまとめています。

---

## 3D/CAD用語

### メッシュ関連
- **メッシュ (Mesh)**: 3D形状を「頂点（点）」と「面（三角形）」の集合で表したもの。
- **水密メッシュ (Watertight mesh)**: 穴や隙間がなく、完全に閉じた立体メッシュ。3Dプリントには必須。
- **頂点 (Vertex)**: メッシュを構成する点。3D座標（X, Y, Z）を持つ。
- **面 (Face)**: メッシュを構成する三角形。3つの頂点で定義される。
- **エッジ (Edge)**: 面と面を繋ぐ線。2つの頂点で定義される。
- **非多様体 (Non-manifold)**: 形状が不正につながっていて、ソリッドとして扱えない状態。
- **内部自己干渉 (Internal self-intersection)**: 形状の内部で面が重なっている状態。

### 形状操作
- **ブーリアン演算 (Boolean operation)**: 2つの立体を組み合わせる演算。
  - **結合 (Union)**: 2つの立体を合体し、1つの立体にする処理。
  - **減算 (Subtraction)**: 1つの立体から別の立体を引く処理。
  - **交差 (Intersection)**: 2つの立体の重なる部分だけを残す処理。
- **輪郭 (Outline/Contour)**: XY平面上の足形の外周境界。
- **凸包 (Convex hull)**: すべての点を包む最小の凸な外形。
- **凹形状 (Concave shape)**: 内側にへこみがある形。凸包では正しく表現できない。
- **高さマップ (Height map)**: 各 (X, Y) に対する Z 高さを与える関数/表。
- **サンプリング (Sampling)**: 連続的な形状から一定間隔で点を抜き出すこと。
- **グリッドサンプリング (Grid sampling)**: 規則的なXY格子に沿ってサンプリングすること。
- **補間 (Interpolation)**: サンプリング点の間の値を推定して滑らかにつなぐ処理。

### ラティス構造
- **ラティス (Lattice)**: 繰り返し構造の内部格子。軽量化や剛性調整に使う。
- **スキン (Skin)**: ラティスの外側を覆う薄い表層。
- **ストラット (Strut)**: ラティス構造を構成する棒状の要素。
- **ユニットセル (Unit cell)**: ラティス構造の基本単位。これを繰り返して全体を構成。
- **格子密度 (Lattice density)**: 格子の細かさ/粗さ。セルが小さいほど密度が高い。
- **通気穴 (Ventilation)**: スキンに開ける穴。通気性を高めるための加工。

## 3Dプリント用語

### ファイル形式
- **STL (STereoLithography)**: 3Dプリント向けのメッシュファイル形式。最も一般的。
- **GLB (GL Binary)**: 3D表示向けのバイナリ形式（glTF）。Web表示に適している。
- **G-code**: 3Dプリンターに送る命令コード。スライサーがSTLから生成する。

### プリント設定
- **スライサー (Slicer)**: STLファイルをG-codeに変換するソフトウェア。
- **レイヤー (Layer)**: 3Dプリントで1層ずつ積み重ねる層。通常0.1〜0.3mm。
- **インフィル (Infill)**: 内部の充填パターン。ラティス構造もインフィルの一種。
- **サポート (Support)**: 浮いた部分を支える構造。後で取り外す。
- **ラフト (Raft)**: 底面に敷く土台。接着性を高める。
- **ブリム (Brim)**: 底面の周囲に張り出す部分。反り防止。

## シェル・OS用語

- **シェル (Shell)**: コマンドラインでOSを操作するインターフェース。
- **Bash (Bourne Again Shell)**: Unix/Linux系OSで使う標準的なシェル。コマンドライン操作とスクリプト実行が可能。
- **PowerShell**: Windows標準のシェル。Bashとは異なるが、同様の操作が可能。
- **WSL (Windows Subsystem for Linux)**: Windows上でLinux環境を実行する機能。
- **Git Bash**: Git for Windowsに付属するBash環境。WindowsでもBashコマンドが使える。
- **コマンドライン (Command line)**: テキストでコマンドを入力して操作する方法。
- **ターミナル (Terminal)**: コマンドラインを実行するアプリケーション。

## フロントエンド用語

### フレームワーク・ライブラリ
- **Next.js**: ReactベースのWebアプリケーションフレームワーク。サーバーサイドレンダリング（SSR）や静的サイト生成（SSG）に対応。
- **React**: UI構築のためのJavaScriptライブラリ。コンポーネントベースで再利用可能なUIを作成。
- **TypeScript**: JavaScriptに型システムを追加した言語。エラーを早期発見し、コードの保守性を向上。
- **Tailwind CSS**: ユーティリティファーストのCSSフレームワーク。クラス名でスタイルを指定。

### 状態管理・UIライブラリ
- **Zustand**: 軽量なReact状態管理ライブラリ。グローバル状態を簡単に管理。
- **Radix UI**: アクセシブルなUIコンポーネントライブラリ。ボタン、スライダー、ダイアログなどを提供。
- **Three.js**: Web上で3Dグラフィックスを描画するJavaScriptライブラリ。
- **React Three Fiber**: Three.jsをReactコンポーネントとして使えるようにするライブラリ。
- **Drei**: React Three Fiber用の便利なヘルパーライブラリ。

### フロントエンド概念
- **コンポーネント (Component)**: UIの再利用可能な部品。Reactでは関数やクラスで定義。
- **フック (Hook)**: Reactの機能。関数コンポーネントで状態やライフサイクルを扱える。
- **状態 (State)**: コンポーネントが持つデータ。変更されると再レンダリングされる。
- **プロップス (Props)**: 親コンポーネントから子コンポーネントに渡すデータ。
- **レンダリング (Rendering)**: コンポーネントを画面に表示すること。
- **クライアントサイド (Client-side)**: ブラウザ側で実行される処理。
- **サーバーサイド (Server-side)**: サーバー側で実行される処理。
- **SSR (Server-Side Rendering)**: サーバー側でHTMLを生成して送信する方式。
- **SPA (Single Page Application)**: 1つのHTMLページで動作するアプリケーション。

## バックエンド用語

### 言語・フレームワーク
- **Python**: 汎用プログラミング言語。科学計算やWeb開発に広く使われる。
- **Streamlit**: PythonでWebアプリを簡単に作れるフレームワーク。データサイエンス向け。

### ライブラリ
- **trimesh**: 3Dメッシュ処理のためのPythonライブラリ。メッシュの読み込み、操作、出力が可能。
- **numpy**: 数値計算のためのPythonライブラリ。配列操作や数学演算を高速に実行。
- **pandas**: データ分析のためのPythonライブラリ。表形式データ（DataFrame）を扱う。
- **plotly**: インタラクティブなグラフを作成するPython/JavaScriptライブラリ。
- **shapely**: 2D幾何学操作のためのPythonライブラリ。ポリゴンの操作に使用。
- **scipy**: 科学計算のためのPythonライブラリ。補間や最適化など。

## IT・コーディング用語

### 基本概念
- **モジュール (Module)**: 機能をまとめたコードファイル。`import` で読み込む。
- **関数 (Function)**: 入力から出力を返す処理のまとまり。
- **引数 (Argument/Parameter)**: 関数に渡す値。
- **クラス (Class)**: データと処理をひとまとめにした設計単位。
- **メソッド (Method)**: クラスに属する関数。
- **スクリプト (Script)**: 単体で実行されるコードファイル。
- **エントリーポイント (Entry point)**: 実行開始地点。`if __name__ == "__main__":` など。

### 依存関係・パッケージ
- **依存関係 (Dependency)**: 動作に必要な外部ライブラリやパッケージ。
- **ライブラリ (Library)**: 再利用可能な機能群。
- **パッケージ (Package)**: 配布可能なライブラリの集合。
- **npm**: Node.jsのパッケージマネージャー。JavaScriptライブラリを管理。
- **pip**: Pythonのパッケージマネージャー。Pythonライブラリを管理。
- **package.json**: Node.jsプロジェクトの依存関係を定義するファイル。
- **requirements.txt**: Pythonプロジェクトの依存関係を定義するファイル。

### API・通信
- **API (Application Programming Interface)**: 機能を外部から使うための呼び出し規約。
- **REST API**: HTTPメソッド（GET、POSTなど）を使ったAPI設計方式。
- **エンドポイント (Endpoint)**: APIの特定の機能にアクセスするURL。
- **HTTP**: Web通信のプロトコル。GET、POST、PUT、DELETEなどのメソッドがある。
- **JSON**: 設定やデータを表すテキスト形式。JavaScript Object Notationの略。

### ファイル形式
- **CSV (Comma-Separated Values)**: 表形式のデータを保存するテキスト形式。カンマ区切り。
- **JSON**: 設定やデータを表すテキスト形式。
- **YAML**: 設定ファイルに使われる読みやすい形式。

### 開発ツール
- **CLI (Command Line Interface)**: コマンドラインで操作する実行方法。
- **UI (User Interface)**: 画面上の操作インターフェース。
- **GUI (Graphical User Interface)**: グラフィカルな操作インターフェース。
- **IDE (Integrated Development Environment)**: 統合開発環境。コード編集、デバッグ、実行を統合。
- **エディタ (Editor)**: コードを編集するソフトウェア。
- **デバッガー (Debugger)**: バグを調査するためのツール。

### 設定・環境
- **設定 (Config)**: 動作条件を決める値の集合。
- **環境変数 (Environment variable)**: システムやアプリケーションの設定値。
- **開発環境 (Development environment)**: 開発用の環境。
- **本番環境 (Production environment)**: 実際にサービスを提供する環境。

### エラー・デバッグ
- **デバッグ (Debug)**: バグの原因を調べて修正する作業。
- **バグ (Bug)**: 期待と違う動作や不具合。
- **例外 (Exception)**: 実行中に発生する予期しないエラー。
- **エラー (Error)**: 処理が失敗した状態やその通知。
- **スタックトレース (Stack trace)**: エラー発生までの関数呼び出し履歴。
- **ログ (Log)**: 処理状況を記録するメッセージ。
- **ログレベル (Log level)**: ログの重要度（例: DEBUG/INFO/WARN/ERROR）。

### テスト
- **テスト (Test)**: 動作が正しいか確認する実行。
- **ユニットテスト (Unit test)**: 小さな単位での自動テスト。
- **統合テスト (Integration test)**: 複数部品を組み合わせたテスト。
- **E2Eテスト (End-to-End test)**: 全体の動作を確認するテスト。

### 開発プロセス
- **リファクタリング (Refactoring)**: 動作を変えずにコードを整理する作業。
- **ビルド (Build)**: 実行可能な形にまとめる工程。
- **デプロイ (Deploy)**: 実行環境へ配置・公開する作業。
- **バージョン管理 (Version control)**: コードの変更履歴を管理する仕組み。
- **Git**: バージョン管理システム。コードの変更履歴を追跡。
- **GitHub**: Gitリポジトリをホスティングするサービス。

### その他
- **エクスポート (Export)**: データやファイルを出力すること。
- **インポート (Import)**: データやファイルを読み込むこと。
- **補完 (Completion)**: 入力中に候補を出して続きを補う機能。
- **型 (Type)**: データの種類（文字列、数値、真偽値など）。TypeScriptでは重要。
- **型推論 (Type inference)**: コンパイラが自動的に型を推測すること。
- **型アノテーション (Type annotation)**: 明示的に型を指定すること。

## MasaCAD固有の用語

### コアファイルの呼び方

> **重要**: ファイル名と実際の用途が異なるため、以下の呼び方を統一してください。

| ファイル名 | 呼び方 | 用途 | 状態 |
|-----------|--------|------|------|
| `core/geometry_v4_frontend.py` | **バックエンド / 本番用** | FastAPI + Next.js で使用中 | ✅ **現在使用中** |
| `core/geometry_v4.py` | **開発用 / Dev版** | 旧Streamlit UI用（非推奨） | ❌ **変更不要** |

#### 注意事項
- **`geometry_v4_frontend.py`** は名前に「frontend」とあるが、実際には **FastAPIバックエンドで使用されている本番コード**
- 「frontend」という名前は「Next.jsフロントエンドの座標系に合わせた」という意味
- 今後の変更・改善は **`geometry_v4_frontend.py`（バックエンド）** に適用する
- **`geometry_v4.py`（開発用）** は旧Streamlit UI用で、今後は変更しない
